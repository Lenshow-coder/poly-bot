## System Overview

This is a **market making bot** for Polymarket prediction markets. Market making means continuously posting both a buy (bid) and a sell (ask) order on each side of a market, profiting from the spread between them. The bot also earns Polymarket's **maker rewards** (paid in USDC) for providing liquidity.

The bot runs on the **Polygon blockchain** and interacts with Polymarket via:
- The **CLOB (Central Limit Order Book) API** for order management
- **WebSockets** for real-time market and user event streaming
- **Web3/RPC** for on-chain operations (balance checks, position merging)
- **Google Sheets API** for configuration and monitoring

## Account & Authentication Architecture

### How Polymarket accounts work

Polymarket doesn't use traditional usernames/passwords. Every account is backed by a **Gnosis Safe smart contract wallet** on Polygon. There are two signup paths:

1. **Wallet connection** — you connect MetaMask or similar; your existing EOA becomes the Safe owner
2. **Email signup** — Polymarket creates a fresh key pair via **Magic.Link** (an embedded wallet provider). This key is completely separate from any external wallet.

### The two credentials

| Credential | What it is | Purpose |
|---|---|---|
| `PK` | Private key of the Safe's owner EOA | Signs API key derivation, signs orders, signs merge transactions |
| `BROWSER_ADDRESS` | Address of the Gnosis Safe contract | Query target for on-chain balances, positions, and portfolio value; Safe contract for merge routing |

Both are always required. They are two aspects of the same access — PK is the owner key, BROWSER_ADDRESS is the Safe it owns.

### Security: fund isolation

If you signed up via **email** (Magic.Link path), the PK is a **fresh key** generated by Magic.Link — it has no connection to any external wallet you may have used to deposit funds. A crypto transfer is one-way; the sender's key is never shared with the recipient. So providing the PK to this bot only gives access to the Polymarket account, not to any external wallet.

If you signed up by **connecting a wallet**, the PK is your main wallet's private key and controls both the EOA and the Safe. To isolate risk, create a dedicated wallet just for Polymarket.

### On-chain approvals

The `approveContracts()` function in `data_updater/trading_utils.py` handles initial setup by approving USDC spending and CTF operator permissions for three Polymarket contract addresses. The README notes that doing at least one trade through the UI also sets these up.

### Signature type

The bot uses `signature_type=2` in the ClobClient, which is Polymarket's "proxy wallet" flow — signing as a Safe owner.

## Google Sheets Configuration

Google Sheets is the bot's entire configuration and monitoring layer.

### Authentication

- Requires a **Google Service Account** with `credentials.json` in the project root
- Uses `gspread` library with Drive + Sheets API scopes
- Fallback: `ReadOnlySpreadsheet` class can fetch public sheets via CSV export without credentials
- The `SPREADSHEET_URL` env var points to the spreadsheet

### Sheet purposes

| Sheet | Written by | Read by | Purpose |
|---|---|---|---|
| **Selected Markets** | You (manually) | `main.py` | Markets to actively trade. Pick from Volatility Markets, assign a `param_type` |
| **All Markets** | `update_markets.py` | `main.py` | Every qualifying market with reward rates, spreads, volatility. Sorted by `gm_reward_per_100` |
| **Volatility Markets** | `update_markets.py` | You | Subset of All Markets with `volatility_sum < 20` — low-volatility "safe" markets. Your shopping list |
| **Full Markets** | `update_markets.py` | `update_stats.py` | Raw market data with all token IDs for position-to-market mapping |
| **Hyperparameters** | You (manually) | `main.py` | Grouped parameter sets keyed by `type` column |
| **Summary** | `update_stats.py` | You | Dashboard: current positions, open orders, earnings per market |

### Key columns in Selected/All Markets

- `question` — human-readable market name (join key between sheets)
- `token1`, `token2` — token IDs for YES and NO outcomes
- `condition_id` — market identifier for trading and merging
- `tick_size` — minimum price increment (e.g., 0.01)
- `min_size` — minimum order size to qualify for rewards
- `trade_size` — standard order size the bot quotes
- `max_size` — maximum position the bot will build per token
- `max_spread` — maximum spread from midpoint where orders earn rewards
- `neg_risk` — "TRUE"/"FALSE" — determines which smart contract path
- `param_type` — which Hyperparameters group to use for risk management
- `3_hour` — 3-hour annualized volatility (risk gate)
- `best_bid`, `best_ask` — reference prices from last sheet update (sanity check)
- `multiplier` — optional buy-size multiplier for cheap tokens (price < 0.10)

### Hyperparameters

Grouped by `type` column. Each group contains:
- `stop_loss_threshold` — PnL % below which stop-loss triggers
- `take_profit_threshold` — PnL % above avg price for take-profit sell orders
- `spread_threshold` — maximum spread for stop-loss execution
- `volatility_threshold` — 3-hour volatility above which trading is paused
- `sleep_period` — hours to pause buying after a stop-loss event

## Startup Sequence

`main.py` orchestrates startup:

1. **`PolymarketClient()`** — authenticates with CLOB API, derives API keys, connects to Polygon via Web3, initializes contract interfaces (USDC, NegRiskAdapter, ConditionalTokens)
2. **`update_once()`** — fetches initial state:
   - `update_markets()` — market config from Google Sheets, populates `REVERSE_TOKENS` mapping, initializes `performing` sets
   - `update_positions()` — current positions from Polymarket data API
   - `update_orders()` — current open orders from CLOB API
3. **Background thread** (`update_periodically`) — polls every 5s for positions/orders, every 30s for market config, prunes stale `performing` entries (>15s)
4. **Two concurrent WebSockets** via `asyncio.gather`:
   - Market WS — streams live order book updates
   - User WS — streams own order/trade events (authenticated with API credentials)

Both WebSocket connections auto-reconnect on failure (5s delay).

## Market Structure: YES vs NO Tokens

Every Polymarket binary market has two outcome tokens:
- **token1** (YES) — price represents probability of the event occurring
- **token2** (NO) — price ≈ 1 - YES price

`global_state.REVERSE_TOKENS` maps each token to its counterpart.

For token2, all prices are **inverted** in `get_best_bid_ask_deets`: bids/asks are flipped via `1 - price`. This means the bot internally always reasons in YES-token price space.

## Live Order Book Management

### Storage (`global_state.all_data`)

```python
all_data[market] = {
    'asset_id': token_id,  # YES token ID
    'bids': SortedDict({0.52: 300, 0.51: 150, ...}),
    'asks': SortedDict({0.54: 200, 0.55: 400, ...})
}
```

Uses `SortedDict` from `sortedcontainers` for efficient price-level lookups.

### WebSocket events (`data_processing.py`)

- **`book`** — full snapshot, replaces entire order book
- **`price_change`** — incremental update at a specific price level (size=0 means removal)
- Only processes updates for the YES token (`asset_id` check) to prevent duplicate processing

Every update immediately triggers `asyncio.create_task(perform_trade(market))`.

## Trading Logic Deep Dive

The core is `perform_trade()` in `trading.py`. It runs on every order book update, uses a **per-market asyncio Lock** to prevent concurrent execution.

### Step 1: Position Merging

If both YES and NO positions ≥ `MIN_MERGE_SIZE` (20), merge them on-chain to recover USDC. Checks local state first, then verifies with on-chain balances before executing.

### Step 2: Price Discovery (`trading_utils.py`)

**`find_best_price_with_size(book, min_size)`** walks the order book to find the best price level with at least `min_size` volume. Returns:
- `best_bid`/`best_ask` — best price with enough liquidity
- `second_best_bid`/`second_best_ask` — next level
- `top_bid`/`top_ask` — absolute best price regardless of size

Falls back to `min_size=20` if initial search with `min_size=100` returns None.

**`get_best_bid_ask_deets`** also computes:
- `mid_price` = (best_bid + best_ask) / 2
- `bid_sum_within_n_percent` / `ask_sum_within_n_percent` — total liquidity within threshold of mid, used as a sentiment ratio

### Step 3: Order Price Calculation (`get_order_prices`)

Strategy: be at the **inside of the spread** (best bid + 1 tick, best ask - 1 tick). But:
- If best bid/ask has insufficient size (< `min_size * 1.5` for bids, < 375 for asks), **match** that price instead of improving
- Crosses prevented: bid can't exceed best ask, ask can't go below best bid
- If bid == ask, fall back to top_bid and top_ask
- **Critical**: if ask price ≤ average cost, it's floored to average cost (prevents selling at a loss)

### Step 4: Position Sizing (`get_buy_sell_amount`)

- **Below max_size**: quote `trade_size` to buy (capped by remaining room), sell `trade_size` only if position ≥ trade_size
- **At/above max_size**: only sell (progressive exit), stop buying unless total exposure < `max_size * 2`
- Hard cap: never buy if position ≥ 250 USDC
- Minimum order size compliance: if buy_amount is between 70% and 100% of min_size, round up to min_size
- For cheap tokens (price < 0.10): apply optional `multiplier` column to scale up buy size

### Step 5: Order Submission Logic

**Buy orders** — gated by multiple conditions:
1. Not in risk-off cooldown period (after stop-loss)
2. `3_hour` volatility below threshold
3. Price within 0.05 of reference price from sheet
4. No significant opposing position (if holding NO tokens, don't buy more YES)
5. Market sentiment ratio ≥ 0 (more buy volume than sell near mid)

Then only actually submit if:
- Better price available than current order, OR
- Position + open orders < 95% of max_size, OR
- Current order is oversized (> 101% of target)

**Sell orders** have two branches:
- **Stop-loss path**: detailed in Risk Management section below
- **Take-profit / normal exit**: sell price = max(avgPrice * (1 + take_profit_threshold%), ask_price). Updates if current order deviates >2% from target or order size < 97% of position

### Step 6: Order Cancellation Efficiency

Both `send_buy_order` and `send_sell_order` compare new vs existing orders. Only cancel and repost if:
- Price differs > 0.005 (0.5 cents), OR
- Size differs > 10%

This minimizes API calls and order churn. Orders for the entire asset are cancelled (not individual orders) — if multiple orders exist for one side, they're all cancelled and consolidated.

## Position Merging

### Why merge?

When you hold both YES and NO tokens in the same market, 1 YES + 1 NO = 1 USDC. Merging converts opposing positions back to USDC, recovering capital.

### How it works

1. `perform_trade()` detects `min(pos_YES, pos_NO) > MIN_MERGE_SIZE (20)`
2. Verifies with on-chain balance check (`get_raw_position` via ConditionalTokens contract)
3. Calls `merge_positions()` which spawns a **Node.js subprocess** (`poly_merger/merge.js`)
4. The JS script routes through the **Gnosis Safe** (`safe-helpers.js`) to call either:
   - `NegRiskAdapter.mergePositions(conditionId, amount)` — for neg-risk markets
   - `ConditionalTokens.mergePositions(USDC, 0x0, conditionId, [1,2], amount)` — for standard markets
5. After merge, local positions are decremented via `set_position(..., 'SELL', scaled_amt, 0, 'merge')`

## Market Discovery & Reward Calculation

`update_markets.py` runs hourly on a separate machine.

### Pipeline

1. **Fetch all markets** via `client.get_sampling_markets()` (paginated)
2. **Process each market** (multithreaded, 5 workers):
   - Fetch order book, extract best bid/ask/midpoint
   - Compute maker reward rates using Polymarket's reward formula
3. **Calculate volatility** (multithreaded, 2 workers):
   - Fetch 1-minute price history from `clob.polymarket.com/prices-history`
   - Compute log returns and annualized volatility at 1h, 3h, 6h, 12h, 24h, 7d, 14d, 30d windows
   - Formula: `std(log_returns) * sqrt(60 * 24 * 252)`
4. **Rank and filter**:
   - `volatility_sum` = 24h + 7d + 14d volatility
   - Volatility Markets = markets where `volatility_sum < 20`
   - Sorted by `gm_reward_per_100` descending
5. **Write to sheets** (only if > 50 markets found, as a sanity check)

### Reward formula

Polymarket distributes maker rewards based on:
- **Proximity to midpoint**: `S = ((v - s) / v)^2` where `v` = max_spread/100, `s` = |price - midpoint|
- **Weighted size**: `Q = S * size` (your order's weighted contribution)
- **Your share**: `(Q / total_Q) * daily_reward / 2` (divided by 2 for bid/ask sides)

`gm_reward_per_100` = geometric mean of best bid reward and best ask reward per $100 quoted. This is used as the primary market selection metric because it penalizes markets where only one side is profitable.

## Risk Management

### Stop-loss

Triggered in `perform_trade()` when either:
- `pnl < stop_loss_threshold` AND `spread <= spread_threshold` (can exit at reasonable price)
- `3_hour_volatility > volatility_threshold` (market too volatile)

Actions:
1. Sell position at best bid (essentially a market order)
2. Cancel all orders in the market
3. Write `positions/<market>.json` with a `sleep_till` timestamp

### Cooldown period

After stop-loss, the JSON file prevents re-entry. On every buy attempt, the bot checks:
```python
if os.path.isfile(fname):
    if current_time < start_trading_at:
        send_buy = False
```
Duration is controlled by the `sleep_period` hyperparameter (in hours).

### Volatility gate

Independent of stop-loss: if `3_hour` volatility exceeds threshold, all buy orders are cancelled and no new ones placed.

### Price sanity check

Buy orders are rejected if the bot's computed price deviates >= 0.05 from the reference price in the sheet (last hourly update). Prevents trading on stale or anomalous data.

### Opposing position check

If the bot holds a significant position in the opposing token (size > min_size), it won't buy more of the current token. This prevents building up both sides simultaneously (which would just need merging).

### Order price bounds

Buy orders are only placed in the 0.10-0.90 price range. This avoids extreme positions near 0 or 1 where the market is nearly decided.

## Market Making Logic — Plain English Summary

The bot wants two orders sitting on the book at all times for each market: a **bid** (buy) and an **ask** (sell). It profits from the gap between them.

### Pricing: where to place orders

The default is to **step in front** of the competition by one tick (e.g., if best bid is 0.52, bid at 0.53). But:
- If the top-of-book order is tiny (below `min_size * 1.5`), the bot **joins** it rather than improving — small orders aren't real competition.
- If stepping in front would cause a cross (bid ≥ ask), it backs off to the raw top-of-book prices.
- The ask price is never set below the bot's average cost — it won't sell at a loss.

### Sizing: how much to buy/sell

The bot gradually **accumulates** a position up to `max_size`, then switches to selling it off:
- Below max_size: buy `trade_size` per order, only sell if position ≥ trade_size.
- At max_size: stop buying, sell `trade_size` to wind down.
- Hard ceiling of 250 shares regardless of max_size.

### When orders are replaced

The bot does **not** cancel on every tick. It compares the new ideal vs. existing order:
- Price moved > 0.5 cents → cancel and replace
- Size changed > 10% → cancel and replace
- Otherwise → leave it alone

When it does cancel, it cancels **all orders for that token** (both sides) and reposts only the new one, because the API doesn't support amending individual orders.

### When buys are blocked

Even if the math says "buy," these gates can block it:
1. **Cooldown** — recently stop-lossed this market (JSON file with "don't buy until" timestamp)
2. **Volatility** — 3-hour vol exceeds threshold → cancel all buys
3. **Price divergence** — computed price differs ≥ 0.05 from hourly reference → cancel
4. **Opposing position** — holding the other token → don't buy (wait for merge instead)
5. **Sell pressure** — more sell liquidity than buy liquidity near mid → don't buy

After passing all gates, the buy only submits if: a better price appeared, the position needs filling, or the existing order is oversized.

### Two modes of selling

- **Normal (take profit):** ask at `avgPrice + take_profit_%`, or the best ask if that's higher. Updated only if existing order deviates >2%.
- **Panic (stop-loss):** sell at best bid (immediate fill), cancel everything, enter cooldown for configured hours.

### What triggers all this

Every order book update from the WebSocket calls `perform_trade()`. The bot also re-evaluates after its own trades confirm. A per-market lock prevents two updates from placing orders simultaneously.

## How WebSockets Work in This Bot

### What is a WebSocket?

A normal HTTP request is like sending a letter: you ask a question, wait for a reply, done. If you want updates, you send another letter. A **WebSocket** is like picking up a phone: you establish a connection once, and then both sides can send messages at any time over that open line. It stays connected until one side hangs up.

This matters for trading because the order book changes many times per second. Polling via HTTP ("any updates?" every second) would be slow and wasteful. A WebSocket lets Polymarket **push** every change to the bot the instant it happens.

### The two WebSocket connections

The bot maintains two simultaneous WebSocket connections:

**1. Market WebSocket** (`wss://ws-subscriptions-clob.polymarket.com/ws/market`)

- **What it receives:** order book updates for every token the bot is watching
- **How it subscribes:** sends `{"assets_ids": [token1, token2, ...]}` on connect
- **Two event types:**
  - `book` — a full snapshot of all bids and asks (received on first connect and periodically)
  - `price_change` — a single price level changed (e.g., "bid at 0.52 is now 300 shares" or "ask at 0.55 was removed")
- **What the bot does with it:** updates its local order book (`SortedDict` of price→size), then immediately calls `perform_trade()` to decide if it should adjust its own orders

**2. User WebSocket** (`wss://ws-subscriptions-clob.polymarket.com/ws/user`)

- **What it receives:** events about the bot's own orders and trades
- **How it authenticates:** sends API key, secret, and passphrase on connect
- **Two event types:**
  - `trade` — one of the bot's orders was filled (matched, confirmed, mined, or failed)
  - `order` — an order was placed, cancelled, or updated
- **What the bot does with it:** tracks trade lifecycle (MATCHED → CONFIRMED → MINED), updates local position and order state, triggers `perform_trade()` to replenish orders after fills

### Why both are needed

The Market WS tells the bot what everyone else is doing (so it can price competitively). The User WS tells the bot what happened to its own orders (so it can update its position tracking and place new orders after fills).

Without the Market WS, the bot would be blind to price changes. Without the User WS, the bot wouldn't know when its orders fill and would have to poll the API — introducing delays that could lead to stale position data and incorrect order sizing.

### Connection management

Both connections use `ping_interval=5` to send keepalive pings every 5 seconds. If either connection drops (network issue, server restart), the `main.py` loop catches the exception, waits 1 second, and reconnects both via `asyncio.gather`. The reconnection also re-subscribes to all tokens / re-authenticates.

## Key Design Decisions

- **Event-driven**: trading happens on every order book tick via WebSocket, not on a timer
- **Per-market locking**: asyncio Locks prevent double-trading the same market
- **Optimistic position updates**: local state updated immediately on MATCHED events, not waiting for blockchain confirmation
- **Performing set**: tracks in-flight trades to prevent API polling from overwriting optimistic local state. Stale entries (>15s) auto-pruned
- **Conservative order management**: orders only replaced when change is meaningful (>0.5 cent / >10% size)
- **Google Sheets as config**: all hyperparameters and market selection are live-editable without restarting the bot
- **Maker vs taker detection**: User WS handler detects whether bot was maker or taker and corrects token/side assignment accordingly
- **Subprocess for merging**: Node.js called via subprocess for on-chain merge operations (leverages existing Polymarket open-source code)

### Trade lifecycle via User WebSocket

| Status | Action |
|---|---|
| `MATCHED` | Add to `performing` set, update local position optimistically, trigger re-evaluation |
| `CONFIRMED` | Remove from `performing`, trigger re-evaluation to repost orders |
| `MINED` | Remove from `performing` |
| `FAILED` | Call `update_positions()` from API to reset to ground truth |

## Deployment: IP Separation and Rate Limiting

### Why separate IPs matter

`update_markets.py` fetches **every market on Polymarket** (not just your selected ones) — it's a discovery tool. This means hundreds or thousands of API calls in an hourly burst. If this runs from the same IP as `main.py`, it can trigger rate limits that degrade the trading bot's low-latency API access.

`main.py` itself is lightweight on API calls (order placement, cancellation, position queries). Its heavy lifting is done via WebSockets, which are persistent connections and don't count toward API rate limits.

### What each process uses

| Connection type | `main.py` | `update_markets.py` |
|---|---|---|
| Market WebSocket | Yes — connected 24/7 | No |
| User WebSocket | Yes — connected 24/7 | No |
| CLOB API calls | Moderate | Heavy (all markets) |
| Polygon RPC | Light (merge balance checks) | No |

### Options for IP separation

1. **Cheap VPS** ($3-5/mo, Hetzner/DigitalOcean) — simplest option. Deploy `update_markets.py`, `credentials.json`, and `.env` on it. It has its own IP by nature.
2. **Docker container with VPN sidecar** — route the container's traffic through a VPN provider. Your `main.py` runs outside on your normal IP.
3. **Proxy** — set `HTTP_PROXY`/`HTTPS_PROXY` env vars (the `requests` library respects these). May not work for `py-clob-client` calls without verification.

Two Docker containers on the **same machine share the same public IP** — Docker gives containers internal IPs (like 172.17.0.2), but outbound internet traffic exits through the host's IP. So Docker alone doesn't solve this.

### VPS vs residential proxy

A **VPS** has a data center IP — cheap, reliable, full control, but identifiable as non-residential. Fine for calling public APIs like Polymarket's, where they want developers to use it and only rate-limit to prevent abuse.

A **residential proxy** routes through real home internet connections, making traffic look like normal consumer activity. More expensive ($20-100+/mo), adds latency, but very hard for services to block. Only needed when scraping websites that actively block data center IPs (Google, Amazon, sportsbooks) — overkill for Polymarket.

### Geo-restrictions

Whether Polymarket's API enforces country-based geo-restrictions on data center IPs is unclear. The website/UI blocks restricted countries, but the API may or may not check. Safe bet: choose a VPS in a country where Polymarket is clearly allowed (UK, most EU countries).

## WebSockets vs API Calls — General Concepts

### The analogy

**A WebSocket is like a TV channel.** You tune in once, and the broadcaster continuously pushes whatever is happening live. But you can't ask the TV what happened 3 hours ago, and you can't tell the TV to place an order for you.

**An API call is like a phone call.** You call, ask a specific question or request an action, get your answer, and hang up. You can ask for anything the service supports.

### When to use which

| Use a WebSocket when | Use an API call when |
|---|---|
| Data changes frequently (many times per second) | You need to perform an action (create/update/delete) |
| You need it in real-time (latency matters) | You need a one-time snapshot |
| You want a continuous stream you didn't ask for | You need historical data |
| The data is "what's happening right now" | The request is on-demand |

Most real-time applications use both. They're complementary, not competing.

### Who creates WebSockets?

The website does. **You can't "WebSocket into" any website.** The server has to specifically build and expose a WebSocket endpoint. Polymarket chose to because they want bots providing liquidity.

### Internal APIs and WebSockets on websites that don't want bots

When you visit a website like a sportsbook, your browser makes dozens of requests behind the scenes — the JavaScript calls internal API endpoints and possibly WebSocket connections to fetch odds, account data, and live updates. These aren't publicly documented, but they're visible in your browser's developer tools (F12 → Network tab).

People reverse-engineer these to build bots, but sportsbooks actively fight this by:
- Obfuscating endpoints (changing URLs, encrypting payloads)
- Browser fingerprinting (checking for real mouse movements, screen size, plugins)
- Rate-limiting and banning accounts that behave like bots
- Requiring KYC so you can't easily make new accounts

This is fundamentally adversarial — the opposite of Polymarket, which publishes its API and hands you WebSocket endpoints because bots make their platform better.
